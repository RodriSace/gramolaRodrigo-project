FLUJO 1

Registro de un Bar

📋 Referencia Memoria: Figura 13 — Sección "Creación de la cuenta"
🏨 Analogía: Darse de alta en un hotel. Rellenas la ficha, te dan un código temporal por email y, hasta que no lo confirmes, la habitación está cerrada.
Navegador (HTML) ➡️ Angular Component ➡️ Spring Controller ➡️ Spring Service ➡️ MySQL (save) ➡️ Mailtrap (SMTP)
1

El barman rellena el formulario de registro [Paso 1 de la Fig. 13]

El navegador muestra la plantilla HTML del formulario de registro y captura la entrada.

Capa / Rol:Angular HTMLVista del Navegador
Archivo:register.component.html
2

Pulsa "Crear Cuenta" y se delega al servicio Angular [Paso 2 y 3 de la Fig. 13]

El componente de registro en TypeScript captura el click, empaqueta los datos y llama al servicio para enviarlo mediante POST al backend.

Capa / Rol:Angular TSComponente ➡️ Servicio Angular (.ts)
Archivo:register.component.ts ➡️ user.service.ts
Elemento:Método onRegister(event) llama a UserService.register()
Acción HTTP:POST /users/register
3

El backend (Controller) recibe el JSON y compara las dos contraseñas [Paso 3 y 4 de la Fig. 13]

El controlador intercepta la llamada. Si no coinciden las claves, devuelve error HTTP 406. Si coinciden, pasa la petición al Service.

Capa / Rol:Java ControllerControlador Spring Boot
Archivo:UserController.java
Elemento:Método register(@RequestBody Map body) con anotación @PostMapping("/register")
4

El Service comprueba si ya existe el email y genera el UUID [Paso 4 y 5 de la Fig. 13]

Busca el email en MySQL. Si hay un registro fantasma (email registrado pero no confirmado o sin suscripción activa), lo elimina. Crea un objeto User con confirmed = false y le genera un UUID aleatorio.

Capa / Rol:Java ServiceServicio de Negocio + Repositorio MySQL
Archivo:UserService.java ➡️ UserRepository.java
Elemento:Método register(...) hace userRepository.findByEmail() y genera UUID.randomUUID().toString()
5

Guarda el usuario en MySQL y envía email mediante Mailtrap [Paso 6 y 7 de la Fig. 13]

Se guarda el usuario en base de datos. Lee dinámicamente la clave "backend_url" de system_config, arma el link y envía el correo por SMTP.

Capa / Rol:MySQLArchivado en BD + Envío SMTP
Archivo:UserRepository.java (save) ➡️ MailService.java
Elemento:userRepository.save(user) (INSERT) y método mailService.sendEmail()
FLUJO 2

Confirmación de Correo

📋 Referencia Memoria: Figura 13 — Sección "Confirmación del token"
📬 Analogía: Haces clic en el enlace de verificación que te llega. El hotel comprueba tu código y, al verificar que es correcto, te acompaña a la ventanilla de pago.
Mailtrap (Clic link) ➡️ UserController GET ➡️ UserService ➡️ MySQL (Update) ➡️ Redirect /payment Angular
1

El barman hace clic en el enlace del email [Paso 8 y 9 de la Fig. 13]

El enlace del correo apunta directamente a una petición HTTP GET a nuestro servidor Spring Boot.

Capa / Rol:Servidor CorreoEnlace externo en Mailtrap
Ruta:GET /users/confirmToken/{email}?token=UUID [Paso 10 de Fig. 13]
2

El Controller del backend recibe la petición y delega al servicio [Paso 10 de la Fig. 13]

El controlador captura el correo y el token UUID de los parámetros de la URL.

Capa / Rol:Java ControllerControlador Spring Boot
Archivo:UserController.java
Elemento:Método confirmToken(...) anotado con @GetMapping("/confirmToken/{email}")
3

El servicio busca el token y cambia confirmed = true [Paso 11 y 12 de la Fig. 13]

Busca en MySQL si existe un usuario con ese UUID. Si existe, cambia su atributo confirmed a true y actualiza la fila en base de datos.

Capa / Rol:Java ServiceServicio de Negocio + Repositorio
Archivo:UserService.java ➡️ UserRepository.java
Elemento:Llama a confirmAccount(token) ➡️ ejecuta userRepository.findByConfirmationToken()
Base Datos:UPDATE users SET confirmed = true WHERE id = ?
4

El backend redirige al navegador a la pasarela de pago del frontend [Paso 13 de la Fig. 13]

Como es un click directo en navegador, se lee la clave "frontend_url" de system_config y se le redirige pasándole el token en la URL de Angular.

Capa / Rol:Java ControllerRedirección Directa de Navegador
Archivo:UserController.java
Elemento:Método response.sendRedirect(feUrl + "/payment?token=" + token)
FLUJO 3

Pago de la Suscripción

📋 Referencia Memoria: Figura 13 — Secciones "Prepago" y "Pago" (Adaptado a Stripe Checkout)
🏪 Analogía: Comprar entrada online. Eliges plan, vas al TPV del banco (Stripe) a pagar y, al volver, tu cuenta está activa.
Angular /payment ➡️ SubscriptionController ➡️ Stripe Session ➡️ TPV Stripe (Pago) ➡️ Webhook (Ruta A) / Angular Redirect (Ruta B)
1

Angular lee los planes de precios dinámicos desde MySQL [Paso 14 de la Fig. 13]

Al cargar, el componente Angular hace un GET al backend. El controlador lee los planes de precios de MySQL (así se pueden cambiar tarifas sin tocar código) y los devuelve.

Capa / Rol:Angular TSComponente Angular ➡️ Spring Controller
Archivo:subscription.component.ts ➡️ SubscriptionController.java
Elemento:`ngOnInit()` hace GET a `/api/subscriptions/plans` ➡️ llama a getPlans()
Base Datos:subscriptionPlanRepository.findAll() (SELECT * FROM subscription_plans)
2

El barman pulsa en el plan y el backend crea la sesión con Stripe [Paso 15 a 20 de la Fig. 13]

Se envía el ID del plan. El controlador de Spring Boot usa el SDK de Stripe en Java para crear una pasarela de pago segura en modo SUBSCRIPTION (cobro recurrente) y le devuelve la URL del TPV a Angular.

Capa / Rol:Angular TSComponente Angular ➡️ Java Controller ➡️ Stripe SDK
Archivo:subscription.component.ts ➡️ SubscriptionController.java ➡️ StripeService.java
Elemento:selectPlan() hace POST `/api/subscriptions/checkout` ➡️ llama a checkout() ➡️ stripeService.createSubscriptionSession()
3

El usuario paga en Stripe y se activan las dos rutas paralelas (A y B) [Paso 21 a 28 de la Fig. 13]

Se implementa tolerancia a fallos. Dos rutas independientes actualizan la cuenta a `subscriptionActive = true` en MySQL:

RUTA A (Webhook):Stripe WebhookStripe ➡️ POST `/api/webhooks/stripe` ➡️ WebhookController.java ➡️ Valida firma criptográfica y activa en BD. (Seguro por detrás).
RUTA B (Redirect):Angular SuccessStripe redirige a `/payment-success?token=UUID` ➡️ PaymentSuccessComponent hace POST `/users/activate-subscription` ➡️ UserController.java activa en BD. (Instantáneo).
FLUJO 4

Login y Conexión con Spotify (OAuth2)

📋 Referencia Memoria: Figura 25 — Flujos 1 y 2 de OAuth 2.0 (Y Figura 19)
🔑 Analogía: Edificio con dos puertas. Puerta 1: Introduces tu contraseña del bar (login). Puerta 2: Das permiso a Spotify para reproducir música (OAuth2). Si ambas se abren, llegas al reproductor.
LoginComponent ➡️ UserController (Login) ➡️ Spotify Auth (Aceptar) ➡️ CallbackComponent ➡️ SpotiController (Canjear)
1

El barman hace login y se verifican credenciales y estados [Paso 1 a 5 de la Fig. 25]

Se realiza el POST de email/password. El controlador verifica: contraseña correcta, email confirmado y suscripción pagada. Si todo es correcto, devuelve el JSON del bar sin el Client Secret por seguridad.

Capa / Rol:Angular TSComponente Angular ➡️ Spring Controller ➡️ Spring Service
Archivo:login.component.ts ➡️ UserController.java ➡️ UserService.java
Elemento:onLogin() hace POST `/users/login` ➡️ llama a login() [Paso 3 y 4 de Fig. 25]
2

Redirección a Spotify para otorgar permisos de control [Paso 6 a 9 de la Fig. 25]

Angular guarda el `clientId` en el `sessionStorage`. Si no está vinculado, el servicio de Angular construye la URL oficial de autorización con su Client ID y redirige el navegador para que acepte los permisos.

Capa / Rol:Angular TSServicio de Spotify en Angular
Archivo:spoti.service.ts
Elemento:sessionStorage.setItem("clientId", ...) y método interno getToken() (construye URL de autorización)
3

Spotify redirige de vuelta con un código temporal y se limpia la URL [Paso 10 a 12 de Fig. 25]

El navegador vuelve a una pantalla intermedia invisible `/callback?code=XXX`. Angular captura el código y limpia la URL al instante para que nadie pueda reutilizar el código del historial.

Capa / Rol:Angular TSComponente de Retorno (Callback)
Archivo:callback.component.ts
Elemento:`ngOnInit()` lee queryParams ➡️ history.replaceState({}, '', '/callback')
4

El backend canjea el código por tokens y los guarda [Paso 13 a 21 de la Fig. 25]

El componente envía el código temporal al backend. El servicio lee el Client Secret de MySQL (para que nunca salga a internet), hace un POST a Spotify, obtiene el Access Token y Refresh Token y los guarda en MySQL.

Capa / Rol:Java ControllerSpring Controller ➡️ Spring Service ➡️ Spotify API
Archivo:SpotiController.java ➡️ SpotiService.java ➡️ UserRepository.java
Elemento:Llama a GET `/spoti/getAuthorizationToken` [Paso 14] ➡️ getAuthorizationToken() lee `clientSecret` [Paso 15] ➡️ POST a accounts.spotify.com/api/token con Basic Auth [Paso 16] ➡️ guarda tokens en MySQL y `sessionStorage` [Paso 20]
FLUJO 5

El barman pone música de fondo

📋 Referencia Memoria: Sección 3.1.2 Flujo 2 ("Acceso a los recursos de Spotify")
📻 Analogía: Poner un CD en una gramola física. Metes un disco entero en el reproductor (playlist), este las añade a la lista (cola) y empieza a sonar.
MusicComponent ➡️ UserController (update-playback) ➡️ Spotify API (Get Tracks) ➡️ MusicController (load-playlist) ➡️ MySQL (insert paid=false)
1

El componente inicializa el SDK del reproductor y arranca el Polling

ngOnInit inicializa `new Spotify.Player` (SDK) y configura el polling reactivo con un Observable para sincronizar la cola visual con MySQL cada 5 segundos.

Capa / Rol:Angular TSComponente de Música Principal
Archivo:music.component.ts
Elemento:`ngOnInit()` ➡️ initSpotifySDK() y subscripción a `queueService.getQueuePolling()`
2

El barman selecciona una playlist y se guarda como activa en MySQL

El componente notifica de inmediato al backend qué playlist se ha activado para guardarlo en la base de datos.

Capa / Rol:Angular TSComponente Angular ➡️ UserController ➡️ UserService
Archivo:music.component.ts ➡️ UserController.java ➡️ UserService.java
Elemento:selectPlaylist() hace POST `/users/update-playback` ➡️ llama a updatePlayback()
Base Datos:Guarda `currentPlaylistUri` en la fila del bar en MySQL
3

Se descargan los tracks de Spotify y se vuelcan en la cola de MySQL

Angular descarga las canciones, las mapea y hace POST al backend. El backend, bajo una transacción de base de datos, borra la música de fondo antigua e inserta las nuevas canciones con `paid = false` al final.

Capa / Rol:Java ControllerSpring Controller ➡️ Spring Service ➡️ QueueRepository
Archivo:music.component.ts ➡️ MusicController.java ➡️ QueueService.java
Elemento:POST `/api/music/queue/load-playlist` ➡️ loadPlaylist() ➡️ loadPlaylistIntoQueue()
Base Datos:Transaccional: `queueRepository.deleteAll(noPagadas)` ➡️ `queueRepository.save(nuevas)`
4

Se reproduce la canción de la posición 0 de la cola en el SDK

Angular lee la cola, extrae el `spotifyTrackId` de la posición 0 de la lista y llama al SDK de Spotify enviándole el ID del track y el Device ID del altavoz virtual.

Capa / Rol:Angular TSComponente de Música ➡️ Spotify Web SDK
Archivo:music.component.ts
Elemento:playQueuePosition0() llama a spotiService.playTrack()
Acción:PUT https://api.spotify.com/v1/me/player/play
FLUJO 6

Cuando una canción termina

📋 Referencia Memoria: Paso de transición automático y red de polling activo
📼 Analogía: Modo "siguiente pista" automático. El disco termina de sonar, la gramola quita ese tema de la lista, todos los de detrás avanzan un puesto y empieza a sonar el nuevo.
Spotify SDK Event ➡️ MusicComponent (finishCurrentSong) ➡️ MusicController (finish) ➡️ MySQL (Delete & Shift) ➡️ Recargar cola y play
1

Detección de fin de pista por listeners y timer con cerrojo de seguridad

Se implementan 3 sistemas de detección redundantes (cambio de track del SDK, pausa cerca del final o un timer manual segundo a segundo). Se activa `isFinishing = true` para evitar llamadas duplicadas simultáneas.

Capa / Rol:Angular TSDetección en Listeners del SDK
Archivo:music.component.ts
Elemento:Método registerPlayerListeners() ➡️ evento `player_state_changed` ➡️ llama a finishCurrentSong()
2

Angular realiza un shift local instantáneo y avisa al backend

Para un efecto visual inmediato sin retraso de red, Angular borra localmente el primer elemento y hace un POST al backend.

Capa / Rol:Angular TSComponente de Música
Archivo:music.component.ts
Elemento:finishCurrentSong() ejecuta `this.queue.shift()` y hace POST `/api/music/queue/finish`
3

El backend borra la canción de MySQL y desplaza las posiciones de la cola

Bajo una transacción SQL, el servicio de Spring borra el track que acaba de sonar y actualiza la columna `position` de todos los elementos posteriores restándoles 1.

Capa / Rol:Java ControllerSpring Controller ➡️ Spring Service ➡️ QueueRepository
Archivo:MusicController.java ➡️ QueueService.java ➡️ QueueRepository.java
Elemento:Método finishSong() ➡️ removeFirstAndAdvance()
Base Datos:`queueRepository.delete()` (pos 0) ➡️ bucle restando `position - 1` ➡️ `save()`
4

Angular recarga la cola de la base de datos y hace sonar el nuevo track

El subscribe del POST descarga la cola fresca y limpia de MySQL, repinta la pantalla y ejecuta la reproducción de la nueva primera canción.

Capa / Rol:Angular TSComponente de Música
Archivo:music.component.ts
Elemento:Carga cola con queueService.getQueue() ➡️ ejecuta playQueuePosition0() y pone `isFinishing = false`
FLUJO 7

Un cliente paga para colar su canción

📋 Referencia Memoria: Pago único y cola con prioridad de inserción
🎪 Analogía: Pase VIP. Hay gente esperando en la cola normal de fondo, pero si pagas tu entrada VIP, te pones delante de ellos de inmediato, justo después del último cliente VIP que llegó antes.
Buscador Debounce ➡️ MusicComponent.paySong() ➡️ MusicController (pay) ➡️ Stripe Popup (Pagar) ➡️ PaymentSuccess ➡️ Principal (postMessage)
1

Búsqueda reactiva inteligente con filtros RxJS

El buscador del móvil del cliente usa debounceTime(500) para esperar que el cliente acabe de escribir y distinctUntilChanged() para evitar repetir la misma llamada a la API de Spotify.

Capa / Rol:Angular TSComponente de Música (Buscador)
Archivo:music.component.ts ➡️ spoti.service.ts
Elemento:`onSearchInput()` ➡️ debounceTime(500) ➡️ executeSearch(query) hace GET `/api/music/search`
2

Apertura de popup y solicitud de Checkout a Stripe

Para evitar que la música se detenga al cambiar de página, se abre un `window.open` emergente con un spinner y se pide al backend la URL de Stripe en modo PAYMENT (pago único).

Capa / Rol:Angular TSComponente de Música ➡️ Spring Controller ➡️ Stripe SDK
Archivo:music.component.ts ➡️ MusicController.java ➡️ StripeService.java
Elemento:paySong() abre `window.open(...)` ➡️ hace POST `/api/music/pay` ➡️ stripeService.createSongPaymentSession()
3

El cliente paga en Stripe y es redirigido a la pantalla de éxito

El cliente paga en Stripe. Stripe redirige la popup de vuelta a `/payment-success?type=song`. Este componente lee los metadatos de la canción de los queryParams y hace POST al backend.

Capa / Rol:Angular TSComponente de Éxito de Pago (Popup)
Archivo:payment-success.component.ts
Elemento:`ngOnInit()` lee queryParams ➡️ musicService.addToQueue(payload) hace POST `/api/music/queue/add`
4

El backend calcula la posición prioritaria VIP e inserta en MySQL

El servicio calcula la posición: justo detrás de la última canción que sea de pago (`isPaid=true`). Suma 1 a la posición de todas las canciones de fondo que quedan por detrás y guarda.

Capa / Rol:Java ServiceSpring Service + Repositorio
Archivo:MusicController.java ➡️ QueueService.java ➡️ BarSongRepository.java
Elemento:addSongToQueue() ejecuta `findItemsAfterPosition` ➡️ actualiza posiciones en MySQL ➡️ inserta `new QueueItem(isPaid=true)` ➡️ guarda recibo en `BarSong`
5

La popup avisa a la ventana principal con postMessage y se cierra

Para no esperar los 5 segundos del polling, la popup manda un mensaje a la principal diciendo "pago completado", esta recarga visualmente al instante y la popup se cierra.

Capa / Rol:Angular TSVentana Emergente Popup ➡️ Ventana Principal
Archivo:payment-success.component.ts ➡️ music.component.ts
Elemento:window.opener.postMessage('payment-success') ➡️ handlePaymentMessage() escucha y recarga la cola
FLUJO 8

Recuperación de Contraseña

📋 Referencia Memoria: Restablecimiento de claves mediante tokens de un solo uso
🗝️ Analogía: Perder la llave de casa. Llamas al cerrajero (backend), te manda un código temporal por email, vas a la puerta con él (formulario) y creas una nueva llave. El código se destruye inmediatamente tras usarse.
LoginComponent modal ➡️ UserController (forgot-pwd) ➡️ Email Mailtrap (Token reset) ➡️ resetTokenRedirect (Validar) ➡️ ResetPasswordComponent
1

El barman pulsa "¿Has olvidado tu contraseña?" en la pantalla de Login

Se abre una ventana modal flotante en la pantalla. Escribe su email y se delega al servicio Angular.

Capa / Rol:Angular TSComponente de Login
Archivo:login.component.ts
Elemento:Método onForgotPassword() hace POST `/users/forgot-password`
2

El backend genera un UUID de recuperación y le envía el correo

El servicio de Spring genera un código aleatorio UUID, lo persiste en el campo `resetPasswordToken` de la fila de MySQL y le manda un email mediante Mailtrap.

Capa / Rol:Java ControllerSpring Controller ➡️ Spring Service ➡️ MailService
Archivo:UserController.java ➡️ UserService.java ➡️ MailService.java
Elemento:`forgotPassword()` ➡️ generateResetToken() hace `user.setResetPasswordToken(uuid)` y lo guarda en MySQL
3

El usuario abre su correo, hace clic y es redirigido al formulario

El enlace GET apunta al backend. El controlador comprueba si el token UUID existe en MySQL; si es correcto, lo redirige con sendRedirect a `/reset-password?token=UUID` en Angular.

Capa / Rol:Java ControllerSpring Controller ➡️ Redirección a Angular
Archivo:UserController.java ➡️ UserService.java
Elemento:`resetTokenRedirect()` con `@GetMapping("/resetTokenRedirect/{email}")` hace response.sendRedirect(...)
4

El usuario escribe la nueva contraseña y se destruye el token de recuperación

El componente Angular envía un POST con token y la nueva contraseña. El servicio de Spring actualiza la clave en MySQL y pone `resetPasswordToken = null` para inutilizar el enlace.

Capa / Rol:Angular TSComponente de Recuperación ➡️ Spring Controller ➡️ Spring Service
Archivo:reset-password.component.ts ➡️ UserController.java ➡️ UserService.java
Elemento:`onReset()` hace POST `/users/reset-password` ➡️ resetPassword() ➡️ resetPasswordWithToken()
Base Datos:UPDATE users SET password = ?, reset_password_token = NULL
